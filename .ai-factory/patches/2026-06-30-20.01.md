# From-header double-wrapping produced trailing ">" in recipient mail clients

**Date:** 2026-06-30 20:01
**Files:** packages/vsix-extension/src/mail/smtp-client.ts, packages/service/src/core/smtp-client.ts, packages/vsix-extension/src/test/mail/fromHeader.test.ts
**Severity:** high

## Problem

When the user set `mcpMail.fromName` (e.g. "Хай"), recipients (Yandex webmail) displayed the sender name with a trailing `>` — "Хай>" instead of "Хай". The SMTP `From:` header actually sent on the wire was malformed: `Хай <Хай <user@yandex.ru>>`.

## Root Cause

Two layers both composed `Name <email>` from the same `fromName`/`fromAddress`, so the second layer wrapped an already-wrapped string:

1. Caller (`MailService.sendEmail` / `RemoteMailClient.sendEmail`) built `emailOptions.from = "Хай <user@yandex.ru>"`.
2. SMTP client (`smtp-client.ts` `sendMail`) took that `options.from` and prepended `fromName` AGAIN: `fromHeader = fromName ? \`${fromName} <${fromAddr}>\` : fromAddr` where `fromAddr = options.from`. Since `options.from` was already `"Хай <user@yandex.ru>"`, the result became `"Хай <Хай <user@yandex.ru>>"`.

The SMTP client duplicated responsibility that the caller already owned. There was no test asserting the shape of the `from` header, so the regression shipped.

## Solution

Removed the name-composition from BOTH SMTP clients (`packages/vsix-extension/src/mail/smtp-client.ts` and `packages/service/src/core/smtp-client.ts`). The SMTP client now uses `options.from` verbatim (falling back to `fromAddress` / `username`), and the caller is the single source of truth for the `Name <email>` formatting. Added `fromHeader.test.ts` that captures the `from` passed to the mock transporter and asserts: no `<<`/`>>`, exactly one `<` and one `>`, and the caller format helper produces exactly one wrap.

## Prevention

- A low-level transport client should NOT re-apply business logic the caller already performed. Keep formatting in one place.
- Add a unit test that captures the exact `from` field handed to nodemailer — not just the return value. The existing `smtpClient.test.ts` only checked `result.messageId`, which is why the malformed header went unnoticed.
- When two layers both format the same field, add an assertion that the result contains exactly one `<` and one `>`.
