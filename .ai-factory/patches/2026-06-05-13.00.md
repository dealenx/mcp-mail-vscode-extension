# Verbose debug log on remote mail operations

**Date:** 2026-06-05 13:00
**Files:** packages/vsix-extension/src/debug/debugRunner.ts (new), packages/vsix-extension/src/mail/remote-client.ts, packages/vsix-extension/src/mailSidebar.ts, packages/vsix-extension/package.json
**Severity:** medium (missing feature, not a bug)

## Problem

When using remote mode (`mcpMail.sendMode = remote`), users hitting transient HTTP
failures (e.g. "Remote service returned invalid JSON: Bad Gateway") had no way to
inspect what was actually sent and received. The error message truncates the
response body to 200 chars and hides HTTP status codes, network error codes,
and request bodies. Field reports of VPN-related 502/ENETUNREACH/ETIMEDOUT were
impossible to diagnose without manually `curl`-ing the proxy.

The "Проверить подключение" and "Отправить тестовое письмо" commands in the
sidebar offered no way to enable verbose tracing.

## Root Cause

A previous exploration (`aif-explore`) designed a verbose-debug feature with
the following shape:
- `askVerboseLog()` dialog before the action
- `withDebugCapture()` wrapper that registers a sink
- `RemoteMailClient.setDebugCapture(sink)` for capture on/off
- Output: temporary unsaved `.log` tab in editor

The design was agreed on but never implemented. A subsequent `/aif-verify` was
run against a *different* plan (Shared Mailbox Support), so the missing
implementation was not caught. The user noticed it as "logs not showing".

## Solution

Implemented the agreed design with these specifics:

1. **New module** `packages/vsix-extension/src/debug/debugRunner.ts`:
   - `withDebugCapture(command, supportsCapture, action)` — top-level wrapper.
     Asks user via `showInformationMessage` with buttons
     `Да, открыть лог` / `Нет`. Default is No (modal: false, no auto-decline).
   - `createDebugCapture(command)` returns `{log, finish}`. Captures ISO-timestamped
     lines, environment header (VS Code version, workspace, send mode, remote URL,
     Node version, platform), and final duration.
   - `maskSensitive(obj)` — recursive masking of keys: `password`, `Authorization`,
     `authorization`, `token`, `apikey`, `apiKey`, `api_key`.
   - `openDebugLogInEditor(content, command)` — opens as unsaved `language: 'log'`
     document in `viewColumn: Beside` with `preview: true`.

2. **`RemoteMailClient.setDebugCapture(sink | null)`** — opt-in sink via existing
   object identity, no interface change in `IMailService`. `MailService` (local)
   silently no-ops via the `'setDebugCapture' in service` guard at call sites.
   - `request()`: logs `→ METHOD url`, body summary, body bytes.
   - `handleResponse()`: logs `← status statusText`, `duration ms`, `body bytes`,
     `body[0..500]`. Output-channel keeps 200-char truncation for the legacy
     `Error` message; sink gets 500 chars + bytes.
   - `request()` catch block: extracts `error.cause.code/syscall/address/port`
     (Node `Undici` errors have these on `cause`) and emits a dedicated
     `✗ Network error code=...` line so VPN/proxy issues are visible.

3. **`mailSidebar.ts`** — wrapped only the **remote** branches of
   `mcpMail.checkConnection` and `mcpMail.sendTestEmail` in
   `withDebugCapture(...)`. Local branches untouched (no HTTP traffic to log).
   Added `setServiceDebugSink(service, sink)` helper that uses `'setDebugCapture'
   in service` guard so the type is `any` but the call is type-safe at runtime.

4. **No new dependencies, no new settings, no interface changes** — feature is
   fully opt-in via the dialog.

## Prevention

- When a feature is "designed in explore mode", capture the design as a
  `/aif-plan fast` task in the same session, even if implementation is deferred.
  Otherwise a future `/aif-verify` against a different plan can miss the gap.
- Verbose-tracing features should be **opt-in by default** to avoid performance
  overhead and log noise. Make the dialog non-modal and "No" the default answer.
- Always log network `error.cause` (Node `Undici`/`fetch` failures) — they
  contain `code` (e.g. `ENETUNREACH`, `ECONNRESET`, `ENOTFOUND`, `ETIMEDOUT`)
  which is the only way to distinguish VPN DNS hijack from proxy 502 from
  server crash.
