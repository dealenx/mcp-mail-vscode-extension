## Problem

Пользователи не могли задать **отображаемое имя отправителя** (display name) для SMTP.
В обоих режимах (local и remote) поле `From:` всегда уходило как голый email,
например `ivan@example.com` вместо `Иван Петров <ivan@example.com>`.

Реальный сценарий: пользователь отправляет письмо через `mcpMail.sendTestEmail`
или через чат Copilot — получатель видит только адрес, без имени.

## Root Cause

Архитектурное упущение, а не баг:

- `SMTPConfig` поддерживал `fromAddress` (e-mail) с самого начала, но без `fromName`.
- `MailConfig.SMTP` и Zod-схема `/connect` в бэкенде тоже содержали только `fromAddress`.
- Все `from`-сборки шли через упрощённое выражение
  `options.from || this.config.fromAddress || this.config.username` —
  оно никогда не пробовало приклеить имя, потому что имени в конфиге физически не было.
- VSIX `package.json` объявлял `mcpMail.fromAddress`, но не `mcpMail.fromName`.
- `MailConfig.SMTP` в VSIX был без `fromName`, поэтому даже при наличии записи в Settings
  её некуда было пробросить.

## Solution

Минимальное расширение — добавили **опциональный** `fromName` на всю цепочку:

1. **VSIX `package.json`** — задекларировали `mcpMail.fromName`
   (string, default `""`, `order: 11` сразу после `fromAddress`).
2. **VSIX `src/mail/config.ts`** — резолвим `fromName.trim()` и кладём в `MailConfig.SMTP.fromName`.
3. **VSIX `src/mail/smtp-client.ts` + `mailService.ts`** — сборка `"Name <addr>"` локально
   перед `nodemailer.sendMail` (`emailOptions.from = "${fromName} <${addr}>"`).
4. **VSIX `src/mail/remote-client.ts`** — для remote-режима тот же `fromHeader` собирается
   на стороне VSIX и передаётся как `from` в JSON к `/connect` или `/send-email`
   (бэкенд использует `from` как есть).
5. **Backend `SMTPConfig.fromName`** + `ConnectRequest.smtp.fromName` в Zod.
6. **Backend `src/core/smtp-client.ts`** — даже если поле `from` не передано,
   сервер собирает `"Name <addr>"` из своего `smtpConfig`.
7. **Backend `buildRawEmailMessage`** (в обоих роутах send-email/reply-email) —
   в raw `From:` header тоже попадает имя, чтобы копия в Sent отображала то же имя.
8. **Logging** — префикс `[FIX-FROMNAME]` во всех ключевых точках:
   - резолв конфига
   - сборка `fromHeader` перед отправкой
   - `From:` в сохранённой копии Sent

### Ключевая логика сборки `fromHeader`

```ts
const fromAddr = options.from || this.config.fromAddress || this.config.username;
const fromName = this.config.fromName?.trim();
const fromHeader = fromName ? `${fromName} <${fromAddr}>` : fromAddr;
```

Если `fromName` пуст (или trimmed пуст) → fallback на голый email, без изменений.

## Обратная совместимость

- Поле везде `optional`, дефолт — пустая строка.
- Старые клиенты без `fromName` в payload → бэкенд собирает header без имени (текущее поведение).
- Бэкенд-тесты `SMTP schema should accept fromAddress` и
  `should work without fromAddress (backward compat)` продолжают проходить (10/12 pass,
  2 неудачи — network-only tests, требуют поднятого localhost:3000).

## Prevention

Этот класс упущений — «настройка есть в VSIX, но не проброшена в бэкенд» — легко предотвратить:

- **Чек-лист при добавлении настройки**:
  1. Объявить в `package.json` → `configuration.properties`
  2. Резолвить в `mail/config.ts` → пробросить в `MailConfig`
  3. Использовать во всех местах, где сейчас используется `fromAddress`
     (smtp-client, mailService, remote-client, mailSidebar)
  4. Если есть бэкенд — прокинуть в JSON-payload + Zod + remote SMTPConfig + raw header builder
- **Во время code review**: пройти по новому полю grep'ом во всех package'ах.
- **При расширении raw-headers** (`buildRawEmailMessage`) — никогда
  не использовать «голый» адрес, всегда собирать header тем же алгоритмом,
  что и для SMTP-отправки.
- **Префикс логов `[FIX-FROMNAME]`** (или другой issue-id) — оставлен как
  трассировка для будущих регрессий: легко грепнуть.

## Что НЕ сделали (осознанно)

- **Per-request override** `fromName` в роутах `/send-email`/`/reply-email`
  (для ad-hoc писем). Сейчас имя — strictly из настройки VSIX.
  Если нужно — добавить `fromName?: string` в Zod и обработать.
- **RFC 2047 кодирование** для non-ASCII raw header. Большинство современных
  SMTP/IMAP серверов принимают bare UTF-8 в `From:` по RFC 6532 (RFC 5321 + 6531).
  Nodemailer кодирует MIME-заголовки автоматически. Если начнут приходить
  жалобы — добавить `mime-word` кодирование.
