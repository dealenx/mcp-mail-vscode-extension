# CancellationToken ignored causing MCP tool hangs on cancellation

**Date:** 2026-05-13 10:00
**Files:** src/tool.ts, src/cancellation.ts (new), src/mailTools.ts, src/mail/smtp-client.ts, src/mail/mailService.ts, src/test/mail/cancellation.test.ts (new), src/test/mail/smtpClient.test.ts (new)
**Severity:** critical

## Problem

When the MCP Send Mail method hangs (e.g., SMTP server slow/unresponsive), the user cannot cancel the operation. All tool `call()` methods receive a `vscode.CancellationToken` parameter but prefix it with `_` (intentionally unused). When the user or Copilot cancels a tool invocation, the cancellation signal is completely ignored, and the operation continues running — appearing frozen in the VS Code chat UI.

## Root Cause

1. **CancellationToken never checked** — Every tool handler had `_token: vscode.CancellationToken` and never used it. The `Tool.invoke()` base class caught thrown errors but didn't handle cancellation as a distinct case.
2. **No AbortSignal propagation** — Even though the previous fix added a 30-second timeout to `SMTPClient.sendMail`, during those 30 seconds the cancellation signal was ignored.
3. **SMTP transporter not cleaned up on timeout/cancel** — When `sendMail` timed out or was cancelled, the nodemailer transporter continued running in the background, potentially corrupting subsequent sends.
4. **Timer leak** — `setTimeout` in the timeout race wasn't cleared on successful send.
5. **saveSentMailRecord not wrapped in try/catch** — If the local save failed after a successful send, the tool returned an error even though the email was already sent.

## Solution

- Created `createAbortController(token)` helper that bridges `CancellationToken` to `AbortSignal`
- Created `CancellationError` class and `throwIfCancelled(signal)` helper
- Updated `Tool.invoke()` to catch `CancellationError` and return `{ cancelled: true }` JSON
- Updated `MailSendEmailTool` and `MailReplyToEmailTool` to create `AbortController` from token and pass `signal` through the call chain
- Updated `SMTPClient.sendMail()` to accept `AbortSignal`, listen for `abort` event in the Promise.race, and clean up transporter on cancel/timeout
- Updated `MailService.sendEmail()`, `replyToEmail()`, `saveSentMessage()`, and `ensureSMTPConnection()` to accept and check `AbortSignal`
- Added `clearTimeout` in the `finally` block of `sendMail`
- Wrapped `saveSentMailRecord` calls in try/catch so email send success is never lost
- On cancel/timeout, the SMTP transporter is closed and reset to `null`

## Prevention

- Always propagate `CancellationToken` → `AbortSignal` → `throwIfCancelled()` in all async tool handlers
- Never prefix cancellation tokens with `_` — always use them
- Always clean up resources (timers, connections) in `finally` blocks
- For network-bound operations, always race against both timeout AND cancellation
- Wrap non-critical post-send operations (like local record save) in try/catch so they never mask a successful send
