# 502 Bad Gateway during remote send-email ‚Äî proxy edge, not app crash

**Date:** 2026-06-05 15:10
**Files:** packages/service/src/index.ts, packages/service/tests/api.test.ts, packages/vsix-extension/src/mail/remote-client.ts, packages/vsix-extension/package.json
**Severity:** medium (transient user-facing failure, no data loss)

## Problem

User pressed "–û—Ç–ø—Ä–∞–≤–∏—Ç—å —Ç–µ—Å—Ç–æ–≤–æ–µ –ø–∏—Å—å–º–æ" in remote mode. Verbose debug log
captured the exact sequence:

```
14:43:09.330 ‚Üí POST /api/connect         ‚Üí 200 OK, sessionId=482e...
14:43:10.109 ‚Üí POST /api/send-email      ‚Üí 502 Bad Gateway (11 bytes, plain text "Bad Gateway")
14:43:10.872 ‚Üí user sees: "Remote service returned invalid JSON: Bad Gateway"
```

The error message was misleading: it suggested the response was malformed JSON,
but in fact 11 bytes of plain text is the proxy's stock 502 page (Railway edge
or Cloudflare), not the Bun/Hono app. Hono would have returned a proper JSON
500 with an `error` field.

## Root Cause

502 Bad Gateway with HTML/plain-text body of ~11 bytes ("Bad Gateway\n")
is **always** generated by the upstream reverse proxy (Railway edge,
Cloudflare, nginx), never by our Bun/Hono application. Confirmed by:

1. Reading `send-email.ts:99-103` ‚Äî full request is wrapped in try/catch,
   any Hono error returns JSON 500, not 502.
2. Reading `index.ts` ‚Äî no global `app.onError()` middleware existed, but
   also no `process.on('uncaughtException' | 'unhandledRejection')` was
   registered. Bun's default is to **terminate** the process on
   unhandled rejection, which would cause edge to fail with 502 until
   Railway restarts the container.
3. Reproducing the same scenario via `curl` ‚Äî 502 was NOT reproducible.
   Service returned JSON 500 with proper auth-error message instead,
   meaning the container was healthy at probe time.
4. The "perfect" 1ms connect‚Üí200‚Üí700ms‚Üísend-email‚Üí502 pattern in the log
   is the classic signature of a **deploy rollout** (Railway restarts the
   container for a new image, edge briefly serves 502 to in-flight traffic
   while DNS/load-balancer updates).

## Solution

Two complementary fixes (no infra changes needed):

### Server-side (`packages/service`)

1. **Global `app.onError()` middleware** in `index.ts` ‚Äî any error escaping
   a route handler is now caught and returned as JSON `{ error, version }`
   with HTTP 500, instead of bubbling up as unhandled.
2. **`app.notFound()` handler** ‚Äî 404s on `/api/*` now return JSON, not
   plain text.
3. **`process.on('unhandledRejection')` and `'uncaughtException')` loggers**
   ‚Äî process logs the error with timestamp+version+stack but does NOT
   exit. Container stays alive and serves subsequent requests normally.
4. **`SERVICE_VERSION` constant + `startedAt` in `/` and `/health`** ‚Äî easy
   to verify which version is actually deployed.
5. **Unit test updated** to expect the new version string and `startedAt`.

### Client-side (`packages/vsix-extension`)

In `RemoteMailClient.handleResponse`, when body is non-JSON and status is
502/503/504, throw a **friendly, actionable** error message:

> "Remote service is temporarily unavailable (HTTP 502 Bad Gateway). This
> usually means the proxy is restarting or the upstream service is down.
> Please retry in 5‚Äì10 seconds. The email may not have been sent ‚Äî check
> your outbox before retrying to avoid duplicates."

This replaces the cryptic "Remote service returned invalid JSON: Bad Gateway".

## What this fix does NOT address

- **Cold-start latency of 502** during deploy rollouts is an edge
  concern. Railway offers "zero-downtime" deploys but the
  502-window-of-few-100ms still happens to in-flight requests. The only
  full fix is client-side retry ‚Äî see "Future work" below.
- **IMAP save-to-sent failure** can still take down the whole request.
  A future change should fire-and-forget the IMAP `appendMessage` after
  SMTP success, so SMTP delivery is decoupled from IMAP availability.

## Future work / recommendations

1. Add a 1-shot **client-side retry** in `RemoteMailClient.request` for
   502/503/504 responses (wait 2s, retry once). This will hide the
   edge-rollout 502s entirely. The risk is duplicate sends, so only
   enable for **idempotent** endpoints (`/api/status`, `/api/mailboxes`)
   and skip for `/api/send-email` and `/api/reply-email` (those have
   `idempotencyKey`, so server-side dedup should kick in, but worth
   being explicit).
2. Move the `findSentMailbox + imap.saveMessageToFolder` block in
   `send-email.ts:77-88` into a fire-and-forget promise with `.catch()`
   log only. Today it's already in a try/catch, so this is just for
   cleanliness.
3. Add `process.uptime()` and `nodeVersion` to the `/health` response
   for ops dashboards.

## Prevention

- **Always** register `process.on('unhandledRejection' | 'uncaughtException')`
  in Bun/Node services. Default behaviour of crashing is rarely what you
  want in production HTTP servers.
- **Always** add a global `app.onError()` middleware in Hono. Per-route
  try/catch is necessary but not sufficient ‚Äî a missing catch anywhere
  takes down the whole request lifecycle.
- **Never** trust 502/503/504 from upstream as "JSON parse error". Always
  inspect `response.status` first and emit a context-aware message.
- For edge-routed services, plan for the 502-window during deploys:
  client-side retry or graceful user messaging is required.
