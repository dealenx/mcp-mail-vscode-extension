# vsce packaging fails with bun workspaces — Cannot find module 'imap' at runtime

**Date:** 2026-05-28 20:11
**Files:** packages/vsix-extension/scripts/package.mjs, packages/vsix-extension/package.json, .vscodeignore, .gitignore
**Severity:** high

## Problem

When building the .vsix extension, `vsce package` failed because bun workspaces creates non-standard `node_modules` (symlinks + `.bun` directory). Two failure modes:
1. `vsce package --no-dependencies` created a .vsix without `node_modules`, causing `Cannot find module 'imap'` at runtime
2. `vsce package` without flags failed because `npm ls` validation couldn't handle bun's symlinked `.bun` directory structure (`EUNSUPPORTEDPROTOCOL`, extraneous/missing package errors)

## Root Cause

Bun's workspace manager uses symlinks and a `.bun` cache directory that npm's `npm ls` (used by vsce) cannot validate. The vsce tool requires a standard npm-compatible `node_modules` tree. Additionally, the monorepo root `package.json` with `workspaces` caused `EUNSUPPORTEDPROTOCOL` errors because of `workspace:*` protocol in other packages.

## Solution

Created a `scripts/package.mjs` that:
1. Temporarily hides the monorepo root `package.json` (so npm doesn't try to resolve workspace:* deps)
2. Saves the bun `node_modules` to a temp dir outside the package (so tsc doesn't pick up backup files)
3. Runs `npm install` to create proper flat `node_modules`
4. Runs `npx vsce package --allow-missing-repository`
5. Restores everything in a `finally` block (ensures rollback even on failure)

Also removed `@mcp-mail/core` from vsix dependencies (it's not imported at runtime — local copies of imap-client/smtp-client are used instead).

## Prevention

- When using bun workspaces with vsce, always create a packaging script that replaces bun's node_modules with npm's flat structure before packaging
- Never use `--no-dependencies` with vsce unless you bundle all runtime deps (webpack/esbuild)
- Move backup directories outside the package root to avoid tsc picking them up
